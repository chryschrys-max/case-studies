Content & Documents
Document Management
Upload, preview, file limits, OCR, storage
⚠ Friction

File sizes reach 100s MB. 100s TB total storage. No in-app multimedia preview — media embedded in PDFs. High storage cost from absence of archival.

⚠ Friction

Low file size limit forces document splitting on complex cases (up to 1,000 charges). Some OCR capability. TCQ micro-frontend for doc rendering.

⚠ Friction

Low file size limit. No in-app preview — planned multimedia preview platform. PDF-only, limiting complex cases with video/audio evidence.

✓ Functional

Simple cases; 5MB file limit is sufficient. PDF preview via browser. OCR initiative in progress (Tesseract). DB size still manageable.

Templates
Central management of notices, orders, letters
✗ Gap

Templates not standardised, not centrally managed. Changes require vendor intervention. Differences in look and feel across court divisions.

✗ Gap

Separately implemented. Most frequently cited pain point across all court officer sessions. No central template repository or governance.

✗ Gap

Disparate from other systems. Templates for maintenance orders and family-specific notices are complex and not aligned with civil equivalents.

✗ Gap

Separate implementation. CJTS-specific tribunal orders and notices are not aligned with any other system’s templates.

Input Forms
Case filing forms, minute sheets, intake screens
✗ Gap

Intake forms require simplification. Minute sheets tailored to specific court divisions. Changes are difficult when form logic is embedded in source code.

✗ Gap

Separate form implementations. Minute sheets are highly specialised per court type. Cross-court alignment identified as needed for modernisation.

✗ Gap

Family-specific forms (MCA applications, maintenance orders) are complex multi-party forms. Not aligned with civil or criminal intake patterns.

⚠ Needs work

Simpler forms overall. Self-represented filers find forms confusing. No built-in guidance or smart validation. One-page case info view needed.

Search
Case search, cause book, cross-system
✗ Poorly used

Search barely used by practitioners — does not work well. Cause book search is an external-facing use case. No cross-system index. Keyword only.

✓ Meets needs

Users report search largely meets requirements. No cross-system capability. No semantic search. Large volumes of information remain hidden in documents.

✓ Meets needs

Search meets user requirements within the system. No integration with civil or criminal case records.

⚠ Limited

Basic search for own cases only. No need for global search at current volume. No cross-system visibility.

Broadcast & Doc Sync
Courtroom document synchronisation
— Not present

No courtroom document synchronisation capability. In-hearing document sharing handled by separate means.

★ Only system

Only platform with broadcast functionality — allows judge to navigate and all parties follow. Reports occasional instability. Not replicated elsewhere.

— Not present

No broadcast functionality. Family hearings use other document-sharing approaches.

— Not present

Not applicable at current case type and complexity. Judges review uploaded documents independently.

Archival
Long-term file storage, retrieval
✗ Absent

No archival in place. All files on fast-access storage. 100s TB driving high ongoing costs. Glacier considered but dropped due to access delay for cause book searches.

⚠ Partial

Access to SCRIMS (predecessor system) archived cases via a UI. No archival for current cases. High storage costs from volume of active cases.

✗ Absent

No archival capability. Family records need long retention periods due to ongoing proceedings and enforcement requirements.

⚠ Planned

Considering Glacier archival after 10 years. DB still small enough to be manageable. Not urgent but approaching threshold.

Operations & Scheduling
Calendaring
Hearing scheduling, rooms, quotas
⚠ Friction

Separately implemented. Room availability, court holidays, and quotas manually entered. Insufficient sync with other systems for conflict resolution.

⚠ Friction

Same fragmentation. High-volume scheduling (400+ cases/day) amplifies cost of manual reconciliation. Background processes required to resolve conflicts.

⚠ Friction

Family hearings require room booking that intersects with criminal/civil calendars. No shared view. Manual coordination for multi-party hearings.

✓ Functional

DayPilot JS calendar. Simpler scheduling needs. Separate instances for filing, consultations, and hearings. Functionally adequate for current volume.

Workflow & Tasks
Task assignment, case progress tracking
⚠ Inconsistent

Workflow features exist but coverage is uneven. Users who work across multiple courts experience significant variation in how work is assigned and tracked.

✓ Functional

Case queue, task list, appointments calendar. Supports full criminal workflow from intake through hearing and judgment. Most complete workflow implementation.

✓ Functional

Workflow and task assignment features present. Supports multi-party family proceedings including mediators, social workers, and external agencies.

⚠ Basic

Basic case folder and task list. Adequate for simple tribunal cases. Backend user has limited workflow support. Registrar-led case management.

Notifications
Hearing notices, information requests, updates
⚠ Redundant

In-app inbox, email, SMS. Redundant SGTS SMS gateways for internal vs external users. Functions but with unnecessary complexity and maintenance overhead.

⚠ Redundant

Aligned approach overall. Multiple overlapping gateways. High volume of notifications (bail, hearings, judgment) makes gateway consolidation a meaningful efficiency gain.

✓ Functional

Notifications to parties, social workers, and agencies broadly functional. Some redundancy with civil system gateways. Not flagged as urgent.

✓ Functional

Simpler notification needs. Parties notified via in-app and email. No SMS complexity. Functionally adequate for self-service model.

Finance & Data
Payment
Filing fees, fines, bail, subscriptions
⚠ Misaligned

Subscription model for law firms. Payment via central gateway (eNETS). Fee structure not aligned with other systems. Ongoing issues with payment gateway reliability reported.

⚠ Complex

Bail and fine collection via a separate financial system (IFAS). Online and offline payment modes. Fee structures not harmonised. Two systems involved in every payment flow.

⚠ Friction

Payment gateway: eNETS primary. Maintenance orders involve ongoing payment tracking not just point-of-filing fees. Unique family law payment complexity (recurring orders).

✓ Simpler

Filing fees only. eNETS and PayNow. Simpler fee structure than other platforms. No subscription model or ongoing payment tracking needed.

Reporting & Statistics
Self-service reports, dashboards, ad-hoc queries
✗ Manual

Some standard reports built in. Many queries require manual DB access by vendor. Tableau exists but on a small data subset. Lost key power user recently. Inconsistent capability.

⚠ Fragmented

Reporting DB synced via AWS DMS. No single platform. Vendor delivers ad-hoc reports from production DB. Qlik dashboards available but coverage inconsistent.

⚠ Fragmented

Elixir reporting done by vendor. Tableau used directly by statistics team (strongest among courts’ teams). Still no unified data platform.

✗ Minimal

Monthly DB dump for statistics team. No self-service dashboard. Ad-hoc reporting requires vendor involvement. Appropriate to volume but no scalability.

Access & Identity
User Management & Auth
Roles, access, identity federation
✗ Fragmented

SingPass/CorpPass + WOG AD + CAM for internal users. Law firm admin manages user access within portal. Each court division has its own roles. No unified matrix.

⚠ Complex

Multi-level roles: JO, CO, RO, Bailiff, Interpreter, Call Centre, PPO, plus agency roles. Practicing cert info received from Family platform. No shared role definitions.

⚠ Cross-agency

Agency users (MSF, hospitals) use SingPass/CorpPass. Social workers, mediators, and DJOs have separate role definitions. Most complex user matrix of the four.

⚠ Public-facing

SRPs and companies use the same portal with different views. Mediators in back-end. CJTS Pass for public users without SingPass. Fine-grained tribunal-level role assignment.

System A — Civil Litigation

Highest technical debt

Priority gaps:

Document archival (100s TB at risk), template management, intake form simplification, and search. Oldest platform, longest vendor relationship — modernisation here requires the most careful vendor negotiation.

System B — Criminal & Regulatory

Most operationally critical

Priority gaps:

Broadcast capability stabilisation, document management for high-charge cases, calendaring sync (400+ cases/day magnifies every friction), and reporting self-service. Most risk if taken down for modernisation.

System C — Family Justice

Most cross-agency complexity

Priority gaps:

30+ agency integrations, recurring payment tracking, most complex user/role matrix. MCA and care order workflows require multi-party data sharing that no other system manages at equivalent complexity.

System D — Community Tribunals

Best pilot candidate

Priority gaps:

Self-represented user experience, form guidance, workflow support for registrars. Lowest technical risk, highest usability opportunity. Strong pilot candidate — but learnings must be deliberately structured to generalise to other systems.
